# API Design (Software Design Side)

## API Architecture Overview

The system backend exposes a versioned REST API specifically within the `api/v1` namespace for the Next js frontend to consume. The API communicates exclusively using the JSON format for both requests and responses to maintain a clean boundary between the backend and frontend.

## Authentication and Authorization

The API secures endpoints using JSON Web Tokens generated by Devise and devise jwt.

- **Authentication Endpoint:** `POST /api/v1/auth/login`.
- **Process:** The user submits credentials to the login endpoint, Devise validates these credentials, and devise jwt generates a token.
- **Token Usage:** The client application stores the token and must include it in the header of all subsequent requests formatted exactly as `Authorization Bearer token`.
- **Token Security:** Tokens are short lived with a 1 hour expiration and are secured further by a token blacklist upon user logout.

## HTTP Status Codes

The API utilizes standard HTTP status codes to indicate the result of client requests:

- `200 Success`: The request completed successfully.
- `201 Created`: A new resource was successfully created.
- `400 Bad Request`: The request failed due to validation errors.
- `401 Unauthorized`: The request is missing a token or the provided token is invalid.
- `403 Forbidden`: The user has insufficient permissions to perform the requested action based on their role.
- `404 Not Found`: The requested resource does not exist.
- `422 Unprocessable Entity`: The request violates a specific business rule.
- `500 Internal Server Error`: An unexpected system failure occurred.

## Standardized Error Response Format

When an API request fails, the server returns a structured JSON payload to help the client understand the exact issue. The standard error response includes an error object containing a specific code, a human readable message, and detailed context.

## Request Execution Flow

1. The Next js client sends an HTTP request with the JWT in the header.
2. The Rails Router routes the request directly to the API V1 Controller.
3. Pundit checks the Role Based Access Control permissions for the user.
4. The designated Service Object executes the isolated business logic.
5. The ActiveRecord Model runs a database query against PostgreSQL.
6. The database returns the necessary records to the backend.
7. The controller formats the final data as JSON and sends it back to the client.
